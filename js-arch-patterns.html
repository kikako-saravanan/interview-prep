🎯 Pattern Selection Framework (CORE SENIOR SKILL)

The CONTEXT framework transforms you from someone who knows patterns to someone who can make architectural decisions like a senior developer.

🔑 Why Pattern Selection Matters More Than Pattern Knowledge

Any developer can learn to implement patterns. Senior developers distinguish themselves by knowing:

🤔 WHEN to use patterns

Understanding the right context and timing for pattern application

🚫 WHEN NOT to use patterns

Avoiding over-engineering and unnecessary complexity

⚖️ HOW to evaluate trade-offs

Systematically weighing benefits against costs

🔄 HOW patterns evolve

Planning for changing requirements and scaling

🔍 The CONTEXT Framework

A systematic approach to architectural decision-making that considers all 7 relevant factors:

  • C - Constraints (Technical, Business, Organizational)
  • O - Objectives (Goals, Metrics, Trade-offs)
  • N - Nature of Problem (Complexity, Domain, Growth)
  • T - Team Dynamics (Expertise, Size, Structure)
  • E - Evolution Strategy (Short/Medium/Long-term)
  • X - eXternal Factors (Technology, Dependencies, Regulations)
  • T - Timeline & Technical Debt (Deadlines, Quality, Value)

C - Constraints

📱 Technical Constraints
  • Performance requirements (latency ≤ 100ms, throughput ≥ 1000 RPS)
  • Memory limitations (mobile vs desktop)
  • Browser support requirements (IE11, modern browsers)
  • Existing technology stack (React, Vue, vanilla JS)
  • Security requirements (GDPR, HIPAA compliance)
💼 Business Constraints
  • Development timeline (MVP in 3 months vs long-term project)
  • Budget limitations (startup vs enterprise budget)
  • Team skill level (junior, mixed, senior team)
  • Maintenance requirements (long-term support needed)
  • Regulatory compliance needs
🏢 Organizational Constraints
  • Team size and structure (2-person startup vs 100+ person org)
  • Deployment frequency (daily deploys vs quarterly releases)
  • Testing capabilities (manual QA vs automated CI/CD)
  • Documentation standards (minimal docs vs comprehensive)
  • Technology approval processes

O - Objectives

🎯 Primary Goals (Choose 1-2 max)
  • Speed to Market: Get features to users quickly
  • Maintainability: Easy to modify and extend over time
  • Performance: Optimize for speed and efficiency
  • Scalability: Handle growth in users/data/features
  • Reliability: Minimize bugs and downtime
  • Developer Experience: Productive and enjoyable to work with
📊 Success Metrics
  • Code quality metrics (test coverage, complexity scores)
  • Performance benchmarks (load times, memory usage)
  • Developer productivity (features shipped per sprint)
  • Bug rates and resolution times
  • User satisfaction scores
⚖️ Trade-off Priorities
  • Maintainability vs Speed of Development
  • Performance vs Code Simplicity
  • Flexibility vs Consistency
  • Innovation vs Stability

N - Nature of the Problem

🔢 Problem Complexity
  • Simple (CRUD apps): Basic patterns (Module, Observer)
  • Medium (Business apps): Architectural patterns (MVC, Repository)
  • Complex (Enterprise): Advanced patterns (CQRS, Event Sourcing)
🎨 Problem Domain

🌳 Pattern Selection Decision Trees

Use these decision trees to systematically choose appropriate patterns:

🗃️ State Management Decision Tree

Is the state simple and local?
├─ YES → useState/local variables
└─ NO → Is it shared across components?
    ├─ YES → Is the sharing complex?
    │   ├─ YES → Redux/Zustand + middleware
    │   └─ NO → Context API/Prop drilling
    └─ NO → Component state with props

🧩 Component Architecture Decision Tree

Is it a simple presentational component?
├─ YES → Pure functional component
└─ NO → Does it need complex logic?
    ├─ YES → Does logic need reuse?
    │   ├─ YES → Custom hooks/HOC
    │   └─ NO → Component with hooks
    └─ NO → Compound component pattern

🏗️ System Architecture Decision Tree

Is it a greenfield project?
├─ YES → Modern patterns (Clean Architecture)
└─ NO → Is it a legacy system?
    ├─ YES → Strangler Fig pattern
    └─ NO → Incremental refactoring patterns

💾 Data Access Decision Tree

Do you have multiple data sources?
├─ YES → Repository pattern
└─ NO → Is data access complex?
    ├─ YES → Need caching?
    │   ├─ YES → Repository + Cache
    │   └─ NO → Simple Repository
    └─ NO → Direct data access

🚫 Anti-Pattern Recognition Framework

Equally important is knowing what NOT to do. Here are common anti-patterns and how to avoid them:

🔨 Over-Engineering

Symptom: Using complex patterns for simple problems

Example: Implementing full CQRS + Event Sourcing for a basic CRUD app

Root Cause: Trying to demonstrate pattern knowledge

Solution: Start simple, evolve complexity only when needed

🔍 Detection Signs:
  • More boilerplate than business logic
  • Team spending more time on architecture than features
  • New team members taking weeks to understand simple features

🍲 Pattern Soup

Symptom: Using too many patterns without clear reasoning

Example: MVC + Observer + Factory + Strategy all in one small module

Root Cause: Lack of coherent architectural vision

Solution: Choose 1-2 complementary patterns max per module

🔍 Detection Signs:
  • Code that's hard to follow despite good documentation
  • Multiple abstraction layers for simple operations
  • Patterns fighting against each other

⚡ Premature Optimization

Symptom: Choosing performance patterns before measuring bottlenecks

Example: Object pooling for rarely created objects

Root Cause: Assumptions about performance without data

Solution: Measure first, optimize based on real bottlenecks

🔍 Detection Signs:
  • Complex performance optimizations with no measurable benefit
  • Code optimized for theoretical rather than actual usage
  • Performance patterns that make code much harder to maintain

📋 Copy-Paste Architecture

Symptom: Using patterns because "Google/Netflix uses them"

Example: Microservices for a 2-person team

Root Cause: Not considering context differences

Solution: Evaluate patterns against YOUR specific context

🔍 Detection Signs:
  • Architecture that seems overengineered for the problem size
  • Patterns that don't fit the team or organizational structure
  • Solutions that work for others but create problems for you

👑 God Object/Module

Symptom: One class/module doing everything

Example: A single "Utils" module with 50+ unrelated functions

Root Cause: Lack of proper separation of concerns

Solution: Apply Single Responsibility Principle, proper decomposition

🔍 Detection Signs:
  • Files with 500+ lines and multiple responsibilities
  • Classes that change for many different reasons
  • Modules that are difficult to test in isolation

🔗 Singleton Abuse

Symptom: Using Singleton for everything "global"

Example: Multiple singletons creating hidden dependencies

Root Cause: Misunderstanding of dependency management

Solution: Dependency injection, proper module design

🔍 Detection Signs:
  • Difficult to unit test due to global state
  • Hidden dependencies between modules
  • Race conditions in initialization

🏗️ Core Design Patterns

Fundamental patterns that every JavaScript developer should master

Module Pattern
Easy

Encapsulates code in modules with private and public members using closures

✅ Use When:
  • Need private variables and methods
  • Organizing code into logical units
  • Preventing global namespace pollution
  • Creating reusable code libraries
❌ Avoid When:
  • Modern ES6 modules are available
  • Simple scripts without complexity
  • Need for inheritance structures
🎯 Benefits:
  • Encapsulation and privacy
  • Namespace management
  • Reduced global scope pollution
⚠️ Trade-offs:
  • Can be harder to extend
  • Memory usage for closures
  • Testing private methods is difficult
Observer Pattern
Easy

Objects (observers) subscribe to changes in another object (subject)

✅ Use When:
  • Need loose coupling between components
  • One-to-many dependency relationships
  • Event-driven architectures
  • Model-View separation (MVC)
❌ Avoid When:
  • Simple direct communication is sufficient
  • Performance is critical (observer overhead)
  • Debugging complex event chains
🎯 Benefits:
  • Loose coupling between objects
  • Dynamic relationships
  • Broadcast communication
⚠️ Trade-offs:
  • Can lead to memory leaks
  • Difficult to debug event chains
  • Potential performance overhead
Pub/Sub Pattern
Easy

Publishers and subscribers communicate through a central message hub

✅ Use When:
  • Need decoupled communication
  • Multiple subscribers for same events
  • Cross-component messaging
  • Event-driven architectures
🎯 Real-world Examples:
  • React Context API
  • Redux action dispatching
  • DOM event system
  • Node.js EventEmitter
🆚 vs Observer:
  • More decoupled (indirect communication)
  • Central event hub
  • Publishers don't know subscribers
Singleton Pattern
Medium

Ensures only one instance of a class exists globally

✅ Use When:
  • Global configuration objects
  • Logging services
  • Database connections
  • Cache management
❌ Avoid When:
  • Need for testing isolation
  • Multi-threaded environments
  • When dependency injection is better
⚠️ Common Pitfalls:
  • Hidden dependencies
  • Difficult unit testing
  • Global state issues
  • Violates Single Responsibility
💡 Better Alternatives:
  • Dependency injection
  • ES6 modules for singletons
  • Factory pattern with registry
Factory Pattern
Medium

Creates objects without specifying exact classes to instantiate

✅ Use When:
  • Object creation logic is complex
  • Need different object types based on input
  • Want to encapsulate object creation
  • Plugin or extension systems
🎯 Variations:
  • Simple Factory: One method creates objects
  • Factory Method: Subclasses decide what to create
  • Abstract Factory: Families of related objects
🎯 Benefits:
  • Centralized object creation
  • Loose coupling
  • Easy to extend with new types
Command Pattern
Medium

Encapsulates requests as objects, allowing parameterization and queuing

✅ Use When:
  • Undo/Redo functionality needed
  • Queuing operations
  • Logging and auditing actions
  • Macro recording
🎯 Real-world Examples:
  • Text editor undo/redo
  • Database transactions
  • GUI button actions
  • Redux actions
🎯 Components:
  • Command: Encapsulates action
  • Receiver: Performs the work
  • Invoker: Triggers command
  • Client: Creates command objects
Strategy Pattern
Medium

Define family of algorithms and make them interchangeable at runtime

✅ Use When:
  • Multiple ways to perform a task
  • Need to switch algorithms dynamically
  • Avoiding large conditional statements
  • Payment processing systems
🎯 Examples:
  • Sorting algorithms (quicksort, mergesort)
  • Payment methods (credit card, PayPal, crypto)
  • Validation strategies
  • Compression algorithms
🎯 Benefits:
  • Open/Closed principle compliance
  • Runtime algorithm switching
  • Eliminates conditional complexity
Decorator Pattern
Medium

Add new functionality to objects without altering their structure

✅ Use When:
  • Need to extend object behavior dynamically
  • Subclassing is impractical
  • Want composable enhancements
  • Middleware systems
🎯 Examples:
  • Express.js middleware
  • React Higher-Order Components
  • Java IO streams
  • Python decorators (@decorator)
🎯 Benefits:
  • Flexible alternative to subclassing
  • Behavior can be added/removed at runtime
  • Single Responsibility Principle

🏛️ Architectural Patterns

Large-scale patterns for organizing entire applications and defining system structure

MVC (Model-View-Controller)
Medium

Separates application logic into three interconnected components for better organization

✅ Use When:
  • Need clear separation of concerns
  • Multiple views for same data
  • Complex user interactions
  • Team needs defined responsibilities
  • Large applications with business logic
🎯 Components:
  • Model: Data and business logic
  • View: User interface representation
  • Controller: Handles user input and coordinates
🎯 Benefits:
  • Clear separation of concerns
  • Multiple views possible
  • Easier testing and maintenance
  • Team can work on different layers
⚠️ Trade-offs:
  • Can be overkill for simple apps
  • Tight coupling between components
  • Complexity in data flow
  • Learning curve for teams
MVP (Model-View-Presenter)
Medium

Similar to MVC but Presenter handles all UI logic and acts as intermediary

✅ Use When:
  • Need better testability than MVC
  • Complex UI logic
  • Passive view requirements
  • Want to isolate view from model completely
🆚 vs MVC:
  • View doesn't directly observe Model
  • Presenter mediates all communication

Week 1: Setup Redux Toolkit infrastructure 2. Week 2: Migrate cart state to Redux 3. Week 3: Migrate user session state 4. Week 4: Add RTK Query for API calls ## Review Date 2024-12-01 (Review effectiveness after 3 months)

🚀 Accelerated Learning Strategies

Connect Patterns

Understand how patterns work together and when they conflict. Real applications use multiple patterns that must harmonize - learn these relationships.

Action: Map out pattern relationships in your projects and document conflicts.
🤔
Question Everything

Always ask "why this pattern?" and "what are the alternatives?" Senior developers justify their choices with data and reasoning, not just familiarity.

Action: Before using any pattern, write down 3 alternatives and why you chose this one.
📝
Document Decisions

Create Architecture Decision Records (ADRs) for every pattern choice. This develops your decision-making skills and creates a learning trail.

Action: Write an ADR for your next architectural decision using our template.

Practice Explaining

Teach patterns to others or record yourself explaining concepts. Teaching reveals gaps in understanding and improves communication skills.

⚠️ Common Mistakes to Avoid

Learn from others' mistakes and avoid these common pitfalls:

🔨
Pattern Hammer Syndrome

Using the same pattern for every problem because it's familiar. Each problem needs its own solution.

Solution: Always evaluate 3 alternatives before choosing a pattern.
🎯
Premature Pattern Selection

Choosing complex patterns before understanding the actual requirements and constraints.

Solution: Start simple, gather data, then evolve to appropriate complexity.
📚
Pattern Perfectionism

Trying to implement textbook patterns perfectly instead of adapting them to real-world constraints.

Solution: Patterns are guidelines, not rigid rules. Adapt them to your context.
🏢
Ignoring Team Context

Choosing patterns without considering team skills, project timeline, or organizational constraints.

Solution: Use the CONTEXT framework to evaluate all constraints.
